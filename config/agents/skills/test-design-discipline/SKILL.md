# テスト設計の規律

## 基本方針

テストは「実装したから書く」ものではなく、「壊れてほしくない仕様を固定する」ために書く。

依頼された挙動を毎回そのまま回帰テストとして追加しない。まず、そのテストが将来どの誤変更を防ぐのか、既存テストや型で既に守られていないかを確認する。

テスト対象は内部実装ではなく、利用者から見える振る舞い、ドメインルール、不変条件、境界処理、外部システムとの接点にする。テストでは事前条件と事後条件を表し、途中でどの関数が何回呼ばれたかを知りすぎないようにする。

## テスト追加の判断

優先してテストを書く場面:

- 絶対に守るべき仕様がある
- 変更が不安な仕様や壊れやすい境界に触れる
- issue、障害、レビュー指摘、ユーザー報告などで具体的に示された不具合を再現し、再発を防ぎたい
- ドメインルール、不変条件、可視性、操作可能性に関わる
- DB、HTTP、シリアライズ、時刻、並行処理、外部 API などの境界に関わる
- 新しく導入した再利用可能な helper や純粋関数が、複数箇所の判断を支える

テストを足さない判断をしてよい場面:

- 型や constructor によって不正状態が作れない
- 既存テストが同じ仕様を実際の経路で十分に押さえている
- getter、単純な mapper、明白な分岐など、テストしても仕様の保護にならない
- 実装詳細を固定するだけで、将来の改善を不必要に縛る
- 「バグかもしれない」「仕様かもしれない」という推測だけで、守るべき挙動を特定できていない
- 今回のタスクの確認には有効だが、長期的な回帰テストとして残す価値が薄い

テストを追加する場合は、そのテストが将来どの誤変更を防ぐのか説明できるようにする。

## 実際の経路でテストする

守りたい仕様がある場合は、できるだけ実際に使われる経路に沿ってテストする。

- public API、usecase、handler、command、query など、利用者が通る入口から確認する
- 内部 helper だけを直接テストして、実際の経路が壊れている状態を見逃さない
- ただし、純粋関数やドメインルールのように独立した意味を持つ処理は、単体でテストしてよい
- 広すぎる回帰テストをやみくもに増やさず、守る仕様に対して必要な経路を選ぶ

## 実装詳細を知らないテスト

テストは、原則として事前条件と事後条件に集中する。

避けるもの:

- private method や内部 helper の呼び出し順に依存するテスト
- mock に「この関数がこの順番で何回呼ばれ、そのたびにこれを返す」と細かく教えるテスト
- 内部データ構造、分割方法、loop の回数などを固定するテスト
- 実装の都合で変わる snapshot やログ出力への過剰な依存

望ましいもの:

- 入力、事前状態、実行、観測可能な結果を明確にする
- 永続化結果、返り値、公開イベント、外部境界への意味ある出力を確認する
- mock より fake、in-memory 実装、test double、実際の軽量 adapter を優先する
- 外部サービスに依存する場合も、プロトコルや payload の意味を確認し、呼び出し回数だけを目的にしない

## Mock の扱い

mock は外部境界を切り離すための道具であり、内部実装を固定するための道具ではない。

- mock を使う前に、fake や純粋関数への切り出しで確認できないか考える
- mock の期待は、仕様上意味のある外部出力に限定する
- 「何回呼ばれたか」は、それ自体が仕様である場合だけ確認する
- テストが mock の準備で読みにくくなるなら、設計や境界が複雑すぎないか見直す

## セルフレビュー

テストを書いた後、または書かない判断をした後に確認する。

- このテストはどの仕様を守っているか説明できるか
- 実際に使われる経路に近いところで確認できているか
- 事前条件と事後条件が読み取れるか
- 実装詳細、呼び出し順、mock の戻り値設定に依存しすぎていないか
- 型や既存テストで保証済みの性質を重複して確認していないか
- 広すぎる回帰テストや snapshot で将来の改善を縛っていないか
- テストしない判断をした場合、その理由を説明できるか

違和感が残る場合はテスト設計か実装境界を見直し、もう一度このレビューを行う。

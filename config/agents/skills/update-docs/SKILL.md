# ドキュメント作成・更新スキル

## ドキュメントの種別と役割

| ファイル | 対象 | 内容 |
|---|---|---|
| `README.md` | 人間 | セットアップ手順、概要、使い方 |
| `AGENTS.md` | AI Agent | 編集ルール、運用方針、禁止事項 |
| `CLAUDE.md` | Claude Code | `AGENTS.md` へのシンボリックリンク |

`AGENTS.md` と `CLAUDE.md` を別ファイルとして二重管理しない。`CLAUDE.md` が必要な場合は、`AGENTS.md` へのシンボリックリンクとして作成する。

```bash
ln -s AGENTS.md CLAUDE.md
```

## 配置原則

ドキュメントはそのディレクトリ内に置く。あるディレクトリについての説明は、そのディレクトリ内の README または AGENTS.md に書く。上位のドキュメントがサブディレクトリの詳細まで説明するのは避ける。それは各サブディレクトリのドキュメントが担う。

## ツリー構造を書かない

ディレクトリ・ファイルの tree 構造をドキュメントに書いてはいけない。

tree はすぐに腐る。ファイルの追加・削除・リネームのたびに手動更新が必要になり、実態と乖離しやすい。構造を知りたい場合は `ls`/`find`/Glob ツールで実際のファイルシステムを参照すればよい。どうしても構造の概念を説明したい場合は、prose で「〜を管理するディレクトリ」と書くにとどめ、ファイルの列挙はしない。

## 書くべきこと / 書かないこと

**書くべき:**

- なぜこの設計・構成になっているか（理由・背景）
- コードを読んでも自明でない制約・禁止事項
- 変更時に踏むべき非自明なステップ
- 役割・責務の境界（何がどこを正本とするか）

**書かないこと:**

- コードを読めばわかること（"このファイルは〇〇を設定します" 等）
- ファイル・ディレクトリの一覧やツリー構造
- 実装詳細の逐次説明
- 他のドキュメントに書いてある内容の繰り返し

## ドキュメントを更新するタイミング

更新する:

- 新しいディレクトリ・モジュールを追加したとき（そのディレクトリの README/AGENTS.md を追加）
- 設計の意図や制約が変わったとき
- 変更時に「なぜこうなのか」が分かりにくいと感じたとき

更新しない:

- ファイルを追加・削除しただけ（tree 構造は書かないので更新不要）
- コードを見れば自明な変更

## 手順

1. **現状確認**: 対象ディレクトリに既存の README/AGENTS.md があるか確認する
2. **既存を優先**: 既存ファイルがあれば新規作成より更新を優先する
3. **README を書く（人間向け）**: セットアップ・使い方・概要のみ。詳細はリンクで誘導する
4. **AGENTS.md を書く（AI Agent 向け）**: 編集ルール・禁止事項・変更判断基準を記述する
5. **CLAUDE.md をシンボリックリンクとして作成**: AGENTS.md が存在し、Claude Code がそのディレクトリを読む可能性があるなら `ln -s AGENTS.md CLAUDE.md` で作成する
6. **上位ドキュメントの参照確認**: 親ディレクトリの AGENTS.md でそのディレクトリに触れる必要があるか判断する。参照するなら 1 行程度で十分

## スコープ

このスキルは README と AGENTS.md（および CLAUDE.md シンボリックリンク）の作成・更新のみを行う。home-manager や Nix 設定の変更は行わない。

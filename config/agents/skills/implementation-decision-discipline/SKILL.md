# 実装判断の規律

実装タスクを、要求整理から完了報告までフェーズ順に進める。各フェーズには入る条件、やること、出る条件がある。前のフェーズの出る条件を満たさないまま次へ進まない。例外は Phase 2 だけで、Phase 1 と並行して進めてよい。成果物の品質がぶれる主因は、完了の基準を最初に固定しないこと、subagent との受け渡しが自由形式なこと、完了判定を実物照合でなくその場の印象で行うことにある。このフェーズ制はそこを塞ぐ。

## 基本方針

- メインセッションは統括者に徹する。要求整理、計画固定、スコープ判断、検収、最終判断を担い、調査・実装・テスト設計・レビューは subagent に分担する。これを標準の進め方とする。
- 実行環境で使える tool を確認し、subagent 起動用 tool があれば必ず使う。tool discovery 用の tool があれば、それで探す。
- 最低でも実装担当と完了前レビュー担当の 2 体を分ける。同じ subagent に自分の実装をレビューさせない。
- 完了判定と完了報告は、その場の印象ではなく Phase 0 で固定した計画との照合で行う。これがぶれ防止の軸になる。

subagent を使わずにメインだけで進めてよいのは、次をすべて満たす場合に限る。

1. subagent 起動用 tool を探索した。
2. tool 一覧を確認しても起動用 tool がない、または起動に失敗して代替も見つからなかった。
3. どの tool を確認し、どう失敗し、代替の有無をどう確認したかをユーザーに報告した。
4. それでもメインで続行する理由を明示した。

「実行環境により subagent を起動できませんでした」とだけ述べて続けてはならない。確認していないなら、使えないと判断しない。

## Phase 0 計画の固定

**入る条件**: 要求整理と初期調査を終え、変更の中心と候補ファイルが見えた。まだ subagent を起動していない。

**やること**: 設計前提を確定してから、作業計画を確定する。以後の完了判定と完了報告は、すべてこの計画との照合で行う。

### 設計前提の確定

設計メモや要求を決定の集合として読まない。実装 subagent を起動する前に次を行う。

- メモの内容を「決定事項」「保留事項」「今回触らない事項」に分類する
- 保留事項が実装範囲に影響するなら、ユーザーに確認してから実装に入る
- 設計判断（型・境界・責務の変更、新しい概念の導入）を含むタスクでは、再検討 subagent を起動する

再検討 subagent に確認させる 3 つの疑い:

- 前提を疑う: この設計が依拠している前提は、要求と既存コードに照らして成立しているか
- 案を疑う: この案は元の問題を消しているか。名前・場所・構造を変えて、問題を移しただけではないか
- 先送りを疑う: この対応はパッチ的で、同じ問題が別の形で再発する余地を残していないか

再検討 subagent の起動ルール:

- 検討した本人に再検討させない。メインセッションが検討した設計の再検討は、必ず別の subagent に任せる。検討した側は、新しい発見がない限り自分の結論を楽観的に見る。
- 検討の経緯、理由づけ、却下した代替案を渡さない。渡すと結論の追認になる。渡すのは、成果物（設計メモ・設計案）、満たすべき要求、参照すべきコードのパスだけにする。
- 「問題ないか確認して」と依頼しない。「この設計が破綻するとしたらどこか」を問い、反証を探させる。
- ファイルは編集させない。指摘は「成立しない前提 / 移しただけの問題 / 先送りの疑い / 実装前に決めるべき事項」の形式で返させる。
- ドメインモデルに関わる場合は `domain-design-safety-discipline` を使わせる。

### 計画の確定

計画に含める 5 項目:

- 要求の言い換え（1-3 行）
- 設計前提: 決定済みの前提、保留事項、今回触らない事項
- 受け入れ基準: 何が確認できたら完了かを、観察できる条件のリストで書く
- 変更してよい範囲と、触らない範囲
- 検証コマンド一覧: プロジェクトの CLAUDE.md、CI 定義、既存のビルド・テスト設定から特定する

スコープは、差分の小ささではなく要求への過不足のなさで選ぶ。局所対応か根本対応かをここで決める。

- 局所対応: 原因と修正箇所が明確に閉じ、既存の責務境界へ素直に差し込め、範囲を広げるほどリスクが保守性の利得を上回る。
- 根本対応: 同じ不具合が複数箇所に散る、呼び出し側ごとに同じ判断を繰り返す、責務境界のせいで局所対応だと分岐が増える、データ構造・型・API 境界を直すほうが今後の変更を単純にできる、要求が個別症状でなく設計変更を求めている。
- 広げる場合も現在の要求と因果関係がある範囲に限る。関係の薄い改善は混ぜず、別作業に切り出す。既存の設計意図は尊重し、違う書き方を入れるなら理由を持つ。

**出る条件**: 5 項目が埋まった。以降で計画を変える場合は、変更点を明示して更新する。暗黙にゴールをずらさない。

## Phase 1 調査・実装 subagent

**入る条件**: Phase 0 の計画が確定した。Phase 0 を飛ばして、要求理解や参照先の探索を subagent に丸投げしない。

**やること**: 調査・実装 subagent を起動し、下の依頼テンプレートを埋めて渡す。報告フォーマットも指定し、その形式で返させる。

依頼テンプレート（穴埋め）:

- 目的: 要求の言い換えと受け入れ基準（Phase 0 の計画から転記）
- 担当範囲: 変更してよいファイル・範囲と、触らない範囲
- 制約・前提: 既に分かっている制約と、未確認の仮説。事実と仮説を分けて書く
- 参照: メインが既に読んだファイルと、追加で読むべき候補。重い探索を避けるためパスで指定する
- 成果物: 実装差分 / 実装案 / 調査結果のどれか。置き場所（メインのワークツリーへ直接書くのか、別の作業場所か）も指定する
- 使うべきスキルと、その理由
- 報告フォーマット（下記）

テンプレートは各項目を短く埋める。結論を誘導する推測を混ぜず、全会話の要約を丸ごと渡さない。

抽象化の判断基準も渡す。抽象化は、実際の重複・複雑さ・境界の曖昧さを減らすときだけ追加する。

- 望ましい: 不変条件を集約する型、副作用境界を明確にする trait、再利用される純粋なデータ変換、複数箇所で同じ判断が要る helper
- 避ける: 将来使うかもしれないだけの汎用化、呼び出し側の都合を隠すだけの薄い wrapper、既存概念と並列に増える新概念、1 回しか使わないのに理解コストを上げる抽象

報告フォーマット（subagent に必ず返させる）:

- 変更したファイル一覧
- 各変更の要約
- 実行した検証コマンドと結果。実行していないものは「未実行」と明記させる
- 判断に迷った点と、選んだ理由
- やり残しと未確認事項

**出る条件**: 報告フォーマットを満たした報告が返ってきた。

## Phase 2 テスト設計 subagent

**入る条件**: 実装方針が見えた、またはテストを追加しそうになった。このフェーズだけは Phase 1 と並行して進めてよい。実装が全部終わってから起動し、既に書いたテストの追認だけをさせてはならない。

**やること**: テスト設計 subagent を起動する。検証は「変更したから機械的にテストを足す」ではなく、「何が壊れると困るか」を決めてから設計する。

依頼に渡す:

- 守るべき仕様、不安な仕様、壊れやすい境界
- 既存テストで既に守られていること
- テストの追加 / 修正 / 不要判断のどれを検討しているか
- `test-design-discipline` を必ず使うこと。テストを追加するならどの誤変更を防ぐのか、追加しないなら型・既存テスト・ビルド・静的検査で何を担保するのかを答えさせる

テスト設計担当は実装担当やレビュー担当と兼ねてよい。ただしテストの追加・修正・不要判断が関わる場合は、担当が誰であれ必ず `test-design-discipline` を使わせる。

**出る条件**: 変更の中心に対する検証方針が決まり、テストの追加・修正・不要判断の根拠を説明できる。Phase 4 はこの判断を依頼に含めるため、遅くとも Phase 4 に入る前に満たす。

## Phase 3 メインの検収

**入る条件**: 実装 subagent の報告を受け取った。Phase 5 の反復で修正差分ができたときも、ここへ戻って同じ検収を行う。

**やること**: subagent の自己申告だけを根拠に次へ進まない。メインセッション自身が実物と照合する。

- `git status` と `git diff --stat`（成果物の置き場所に応じた同等の手段）で実際の差分を取得し、報告内容（反復での修正なら、修正したはずの内容）と突き合わせる。
- 完了判定に使う検証コマンド（Phase 0 の一覧）を、メインが自分で再実行する。
- 実差分が報告や修正意図と食い違う場合は、修正元（通常は Phase 1 の実装担当）へ差し戻す。

**出る条件**: 直近の変更内容と実差分が一致し、検証コマンドがメイン自身の再実行で通った。

## Phase 4 完了前レビュー subagent

**入る条件**: Phase 3 の検収を通過した。完了報告やコミットの前。

**やること**: 実装担当とは別の subagent にレビューを依頼する。レビュー担当はファイルを編集しない。

依頼に含める（この順に書く。差分と受け入れ基準を先に置き、実装側の判断や不安点は後に置く。実装の経緯を先に読ませると、結論の追認を招く）:

- 最終差分の要約
- 差分の取得方法。差分本文を渡すか、取得コマンドと対象パスを指定する
- 受け入れ基準（Phase 0 の計画から転記）
- 実行済みの検証と結果
- テスト設計 subagent の判断
- 重点的に見てほしい不安点
- レビューだけを行い、ファイルを編集しないこと
- 指摘を次の 3 つに分類すること。各指摘に対象ファイル・該当箇所・理由を付けること
  - MUST FIX: 受け入れ基準の未達、明らかな不具合、検証の失敗、データ破壊やセキュリティ問題
  - SHOULD FIX: 今回の要求と因果関係のある品質改善で、放置しても動作は正しいもの
  - 確認済み: 問題なしと確認できた観点

**出る条件**: 分類済みの指摘が返ってきた。

## Phase 5 反復と完了

**入る条件**: レビュー指摘を受け取った。

**やること**: 指摘した項目だけを直して即完了にしない。修正で別の漏れや設計の歪みが出ていないか、再レビューで確認する。

反復:

- MUST FIX は原則として修正する。修正しない場合は、誤分類・今回の要求外・要求外だが重要なので別作業化・ユーザー確認が必要、のどれに当たるかを明示する。
- SHOULD FIX は、今回の要求と因果関係があり、差分を不必要に広げないものだけ修正する。
- 修正は原則として実装担当 subagent への差し戻しで行う。会話を継続できる仕組みがあればそれを使う。なければ、元の依頼内容に現在の差分と指摘一覧を添えて新規起動する。メインが直接直してよいのは数行程度の軽微な修正に限る。
- 修正後、Phase 3 の検収と Phase 4 のレビューへ戻る。再レビューは、修正箇所と影響範囲を重点にする。
- 新しい MUST FIX が出たら、また繰り返す。

反復を止めてよい条件:

- 未対応の MUST FIX がなく、新しい MUST FIX も出ない。
- 残った SHOULD FIX が、今回の要求に対して過剰か、別作業として扱うべきだと説明できる。
- 同種の SHOULD FIX が繰り返し出ており、直すほど差分が広がると判断できる。

2 回連続で新しい MUST FIX が出る、同種の MUST FIX が再発する、または修正が要求外へ広がる場合は、局所修正を続けない。同種の指摘の再発は、対応がパッチ的で問題を先送りしている兆候として扱う。方針変更・スコープ縮小・ユーザー確認のどれが必要かを決めてから再開する。

意味のある単位が完了したら、ユーザーの指示を待たずに区切りを検討する。中心の実装と検証が終わった、1 つの責務が独立して説明できる、PR や別作業へ移る前、といった場面が区切りどきになる。次の変更がないとビルドやテストが通らない、意図をまだ説明できない、無関係な残骸が混ざる、ユーザーがコミットしない方針を示している場合は、まだ区切らない。区切りとしてコミットするなら、手順・ブランチ確認・メッセージ・署名は `git-commit` スキルに従う。ここでは重複定義しない。

**出る条件**: 反復の停止条件を満たし、下の完了報告を埋めた。「一応動く」「軽く確認した」では完了扱いにしない。

完了報告テンプレート:

- 受け入れ基準それぞれの充足状況
- 最終差分の要約
- 実行した検証コマンドと結果
- 残した指摘とその理由（反復を止めた判断の根拠）

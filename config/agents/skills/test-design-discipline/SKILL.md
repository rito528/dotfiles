# テスト設計の規律

## 基本方針

テストは「実装したから書く」ものではなく、「壊れてほしくない仕様を固定する」ために書く。

依頼された挙動を毎回そのまま回帰テストとして追加しない。まず、そのテストが将来どの誤変更を防ぐのか、既存テストや型で既に守られていないかを確認する。

タスクが正しく行われたかの確認と、将来の仕様を守るテストは別物である。確認は実行、出力や生成物の確認、一時的なスクリプトで行ってよい。テストスイートに残すのは、テスト追加の判断を通ったものだけにする。

テスト対象は内部実装ではなく、利用者から見える振る舞い、ドメインルール、不変条件、境界処理、外部システムとの接点にする。テストでは事前条件と事後条件を表し、途中でどの関数が何回呼ばれたかを知りすぎないようにする。

## テスト追加の判断

優先してテストを書く場面:

- 絶対に守るべき仕様がある
- 変更が不安な仕様や壊れやすい境界に触れる
- issue、障害、レビュー指摘、ユーザー報告などで具体的に示された不具合を再現し、再発を防ぎたい
- ドメインルール、不変条件、可視性、操作可能性に関わる
- DB、HTTP、シリアライズ、時刻、並行処理、外部 API などの境界に関わる
- 新しく導入した再利用可能な helper や純粋関数が、複数箇所の判断を支える

テストを足さない判断をしてよい場面:

- 型や constructor によって不正状態が作れない
- 既存テストが同じ仕様を実際の経路で十分に押さえている
- getter、単純な mapper、明白な分岐など、テストしても仕様の保護にならない
- 実装詳細を固定するだけで、将来の改善を不必要に縛る
- 「バグかもしれない」「仕様かもしれない」という推測だけで、守るべき挙動を特定できていない
- 今回のタスクの確認には有効だが、長期的な回帰テストとして残す価値が薄い

テストを追加する場合は、そのテストが将来どの誤変更を防ぐのか説明できるようにする。

## タスクの写しテストを見分ける

次のシグナルに当てはまるテストは、タスクの確認をテストの形で残そうとしている可能性が高い。書く前に、テスト追加の判断へ戻る。

- 期待値が、依頼内容をそのまま assert に書き写したものになっている
- codegen の出力、スキーマ、設定ファイルなど、他の正本から機械的に導かれる生成物の内容を assert している。その内容は型や生成器が既に保証していないか
- このテストが落ちるのは誰かが意図的にこの変更を戻したときだけで、事故で壊れる経路が存在しない

「利用者から見える」「外部との境界に関わる」に見えても、その内容が型や生成器から機械的に導かれるなら、まず型側の保証で足りないかを確認する。なお、公開 API の後方互換のためにスキーマ全体を契約として検査する仕組み（スキーマ diff の CI チェックなど）は仕様の保護であり、これはフィールドごとの存在 assert をタスクごとに積むこととは別物である。

タスクの確認が必要なら、テストを残す代わりに一時的な手段で検証し、報告では「どう検証したか」「テストとして残すもの」「残さない理由」を分けて述べる。

## 実際の経路でテストする

守りたい仕様がある場合は、できるだけ実際に使われる経路に沿ってテストする。

- public API、usecase、handler、command、query など、利用者が通る入口から確認する
- 内部 helper だけを直接テストして、実際の経路が壊れている状態を見逃さない
- ただし、純粋関数やドメインルールのように独立した意味を持つ処理は、単体でテストしてよい
- 広すぎる回帰テストをやみくもに増やさず、守る仕様に対して必要な経路を選ぶ

## 実装詳細を知らないテスト

テストは、原則として事前条件と事後条件に集中する。

避けるもの:

- private method や内部 helper の呼び出し順に依存するテスト
- mock に「この関数がこの順番で何回呼ばれ、そのたびにこれを返す」と細かく教えるテスト
- 内部データ構造、分割方法、loop の回数などを固定するテスト
- 実装の都合で変わる snapshot やログ出力への過剰な依存

望ましいもの:

- 入力、事前状態、実行、観測可能な結果を明確にする
- 永続化結果、返り値、公開イベント、外部境界への意味ある出力を確認する
- mock より fake、in-memory 実装、test double、実際の軽量 adapter を優先する
- 外部サービスに依存する場合も、プロトコルや payload の意味を確認し、呼び出し回数だけを目的にしない

## Mock の扱い

mock は外部境界を切り離すための道具であり、内部実装を固定するための道具ではない。

- mock を使う前に、fake や純粋関数への切り出しで確認できないか考える
- mock の期待は、仕様上意味のある外部出力に限定する
- 「何回呼ばれたか」は、それ自体が仕様である場合だけ確認する
- テストが mock の準備で読みにくくなるなら、設計や境界が複雑すぎないか見直す

## 差分レビュー

テストを書いた後、または書かない判断をした後に行う。差分と守るべき仕様だけを渡された第三者でも同じ結果になるように、対象を列挙してから、対象ごとの問いに答える。誰がこのレビューを行うかは、このスキルでは決めない。実装の進め方を管理する側に従う。

手順:

1. 対象を列挙する: 追加・変更したテスト / 削除したテスト / テストを書かない判断をした仕様。
2. 対象ごとに下の問いに答える。「問題なし」で済ませず、根拠を 1 行で言い切る。言い切れない対象は、本文の該当節へ戻ってテスト設計から見直す。
3. 問題が見つかった対象は修正し、修正した対象にもう一度この手順を適用する。

追加・変更したテストごとの問い:

- 仕様: このテストはどの仕様を守り、将来どの誤変更を防ぐか
- 経路: 実際に使われる入口（public API、usecase、handler など）を通っているか。単体でテストしているなら、独立した意味を持つ純粋関数やドメインルールか
- 読み取り: 入力、事前状態、実行、観測可能な結果がテストから読み取れるか
- 実装詳細: 呼び出し順、mock の戻り値設定、内部データ構造、snapshot に依存していないか。依存しているなら、それ自体が仕様か
- 重複: 型、constructor、既存テストで既に保証されている性質を繰り返していないか。広すぎる回帰テストで将来の改善を縛っていないか
- 写し: 期待値が今回の依頼内容や差分の写しになっていないか。タスクの確認と仕様の固定を区別したか

削除したテストごとの問い:

- そのテストが守っていた仕様は、何によって引き続き守られるか

テストを書かない判断ごとの問い:

- その仕様は何で担保されるか（型 / 既存テスト / ビルド / 静的検査）を言い切れるか。推測だけで守るべき挙動を特定できていないなら、「テスト追加の判断」へ戻る

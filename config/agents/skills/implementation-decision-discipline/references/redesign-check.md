# 再検討 subagent への依頼

Phase 1 で、設計判断(新しい概念の導入、型・境界・責務の変更)を含むタスクに使う。設計と解釈の両方について反証を探させる。

## 渡すもの

- 要求の原文: Issue 本文やユーザーの依頼文そのもの。メインによる言い換えを渡さない
- 成果物: 設計メモ・設計案
- 参照すべきコードのパス

検討の経緯、理由づけ、却下した代替案は渡さない。渡すと結論の追認になる。

## 確認させる 4 つの疑い

- 解釈を疑う: 要求の原文には他にどんな解釈がありうるか。設計が採用した解釈を裏付ける出典(原文の引用、既存コードの前例)はあるか。出典のない解釈を列挙させる
- 前提を疑う: この設計が依拠している前提は、要求と既存コードに照らして成立しているか
- 案を疑う: この案は元の問題を消しているか。名前・場所・構造を変えて、問題を移しただけではないか
- 先送りを疑う: この対応はパッチ的で、同じ問題が別の形で再発する余地を残していないか

## 起動ルール

- 検討した本人に再検討させない。メインセッションが検討した設計の再検討は、必ず別の subagent に任せる。検討した側は、新しい発見がない限り自分の結論を楽観的に見る。
- 「問題ないか確認して」と依頼しない。「この設計が破綻するとしたらどこか」を問い、反証を探させる。
- ファイルは編集させない。指摘は「出典のない解釈 / 成立しない前提 / 移しただけの問題 / 先送りの疑い / 実装前に決めるべき事項」の形式で返させる。
- ドメインモデルに関わる場合は `domain-design-safety-discipline` を使わせる。

## 指摘の扱い

出典のない解釈が指摘されたら、Phase 0 の解釈ゲートへ戻り、保留として扱い直す。外形に触れるなら、実装に入る前にユーザーへ確認する。

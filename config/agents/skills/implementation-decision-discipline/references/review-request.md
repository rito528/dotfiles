# 完了前レビュー subagent への依頼

Phase 5 で使う。軽量階層のレビュー依頼もこの形式に従う。

依頼に含める項目を、この順に書く。差分と受け入れ基準を先に置き、実装側の判断や不安点は後に置く。実装の経緯を先に読ませると、結論の追認を招く。

- 最終差分の要約
- 差分の取得方法。差分本文を渡すか、取得コマンドと対象パスを指定する
- 受け入れ基準(Phase 1 の計画から転記。軽量階層では簡易版の計画から転記)
- 実行済みの検証と結果
- テスト設計 subagent の判断(軽量階層では省略してよい)
- 重点的に見てほしい不安点
- 置き換えで不要になった旧コード、使われなくなった import・分岐・wrapper が差分に残っていないかの確認。残っていれば、受け入れ基準に削除が含まれるなら MUST FIX、そうでなければ SHOULD FIX に分類すること
- レビューだけを行い、ファイルを編集しないこと
- 指摘を次の 3 つに分類すること。各指摘に対象ファイル・該当箇所・理由を付けること
  - MUST FIX: 受け入れ基準の未達、明らかな不具合、検証の失敗、データ破壊やセキュリティ問題
  - SHOULD FIX: 今回の要求と因果関係のある品質改善で、放置しても動作は正しいもの
  - 確認済み: 問題なしと確認できた観点

差分が型・集約・層境界・public API の設計に触れる場合は、`domain-design-safety-discipline` の差分レビューを使うことも依頼に含める。設定値の変更やドキュメント修正のような、ドメイン設計に触れない差分では含めない。

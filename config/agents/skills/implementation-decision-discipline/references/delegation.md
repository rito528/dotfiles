# 調査・実装 subagent への依頼

Phase 2 で使う。依頼テンプレートを埋めて渡し、報告フォーマットを指定して、その形式で返させる。

## 依頼テンプレート(穴埋め)

- 目的: 解釈ゲートの決定事項と受け入れ基準(Phase 1 の計画から転記)
- 担当範囲: 変更してよいファイル・範囲と、触らない範囲
- 制約・前提: 既に分かっている制約と、未確認の仮説。事実と仮説を分けて書く
- 参照: メインが既に読んだファイルと、追加で読むべき候補。重い探索を避けるためパスで指定する
- 成果物: 実装差分 / 実装案 / 調査結果のどれか。置き場所(メインのワークツリーへ直接書くのか、別の作業場所か)も指定する
- 使うべきスキルと、その理由。実装差分を成果物とする依頼では、`functional-core-style` を既定で含める
- 報告フォーマット(下記)

テンプレートは各項目を短く埋める。結論を誘導する推測を混ぜず、全会話の要約を丸ごと渡さない。起動時の文脈複製オプションも同じ理由で最小にする(SKILL.md の「subagent 起動の共通ルール」を参照)。このテンプレートは、子が親の会話なしで作業を始められるように設計してある。

## 抽象化の判断基準(依頼に含める)

抽象化は、実際の重複・複雑さ・境界の曖昧さを減らすときだけ追加する。

- 望ましい: 不変条件を集約する型、副作用境界を明確にする trait、再利用される純粋なデータ変換、複数箇所で同じ判断が要る helper
- 避ける: 将来使うかもしれないだけの汎用化、呼び出し側の都合を隠すだけの薄い wrapper、既存概念と並列に増える新概念、1 回しか使わないのに理解コストを上げる抽象

## 削除の判断基準(依頼に含める)

置き換えで不要になったコードは、消すところまでが実装に含まれる。履歴は git に残るため、「念のため残す」ことに保険としての価値はない。

消してよいかどうかは、印象ではなく参照の調査で判断する。次をすべて確認できたら消してよい。

- 識別子としての参照を、リポジトリ全体で検索した。参照検索できる tool(LSP など)があれば併用し、なければ grep だけでよい
- 文字列としての参照も検索した。設定ファイル、DI、ルーティング、リフレクション、シリアライズ、テンプレート、CLI 引数からの利用は、識別子の検索に現れない
- 置き換えの場合は、旧経路の呼び出し元をすべて列挙し、それぞれが新経路へ移ったことを確認した
- 削除後に Phase 1 の検証コマンドが通った。ただし、消した経路にテストがなかっただけの可能性があるため、検証が通ることは参照調査の代わりにならない

次のどれかに当たる場合は、消す・残すをその場で決めず、保留事項としてメインへ報告する。

- リポジトリの外から使われうる公開 API、エクスポート、公開スキーマに触れる
- 参照が生成コードや外部の設定にあり、リポジトリ内の検索だけでは使われていないことを証明できない
- 参照はないが、コメントや履歴に意図して残した形跡がある(feature flag で無効化中、移行の途中など)

使われていないと証明しきれないコードを、黙って残さない。残す場合は、その理由を報告に書く。

## 報告フォーマット(subagent に必ず返させる)

- 変更したファイル一覧
- 各変更の要約
- 実行した検証コマンドと結果。実行していないものは「未実行」と明記させる
- 消したコードと、消してよいと判断した根拠(参照をどう調査し、何が見つかったか)。消す判断が絡まなかった場合は「削除なし」と書かせる
- 判断に迷った点と、選んだ理由
- やり残しと未確認事項

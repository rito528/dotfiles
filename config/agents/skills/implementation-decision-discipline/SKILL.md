# 実装判断の規律

## 基本方針

実装タスクでは、最初に動くコードを書いて終わりにしない。

要求の根本原因に対して、過不足のないスコープで実装し、検証と差分レビューで確認する。漏れや違和感があれば修正し、完了扱いにする前にもう一段深く見直す。

まず変更の中心を決める。今回の要求を局所対応で満たせるのか、原因や設計境界まで直すべきなのかを見極める。

この skill は、メインセッションが直接実装を抱え込むための手順ではない。メインセッションは統括者として、要求整理、スコープ判断、subagent への依頼、成果物レビュー、最終判断を担当する。調査、実装、テスト設計、差分レビューは、利用できる subagent に分担することを標準の進め方とする。

## Subagent 利用

この skill を使う実装タスクでは、メインセッションだけで作業を完結しない。まず現在の実行環境で使える tool 一覧を確認し、subagent 起動用 tool が見つかる場合は必ず subagent に作業を分担する。tool discovery 用の tool がある環境では、それを使って subagent 起動用 tool を探す。

メインセッションは、subagent を起動する前に共有コンテキストを作る。各 subagent が同じ情報を取り直さなくて済むように、必要な事実、判断、参照先を短くまとめて渡す。

共有コンテキストに含めるもの:

- ユーザー要求を 1-3 行で言い換えたもの
- 変更してよい範囲と、触らない範囲
- 読んだファイル、関連する既存仕様、入口となる関数や設定
- 既に分かっている制約、未確認の仮説、判断に迷っている点
- 使うべき skill と、その subagent に期待する使い方
- 実行済みの検証、失敗した検証、まだ実行していない検証

共有コンテキストには、結論を誘導する推測を混ぜない。実装案がまだ決まっていない段階では、観測した事実と未確認の仮説を分けて書く。不明な項目は不明と書き、該当しない項目は省略してよい。

### 起動タイミングと役割

少なくとも次の役割を分ける。最低でも、実装担当 subagent と完了前レビュー担当 subagent の 2 体を起動する。

- 調査と実装を行う subagent
- テスト追加、テスト修正、またはテスト不要判断を確認する subagent
- 完了前に、差分、検証、スコープを確認するレビュー subagent

実装担当と完了前レビュー担当は分ける。テスト設計担当は、実装担当またはレビュー担当と兼ねてもよいが、テスト追加、テスト修正、テスト不要判断のいずれかが関わる場合は、必ず `test-design-discipline` を使う。実装担当にテスト変更を任せる場合も、依頼時点で `test-design-discipline` を使うよう明示する。

各 subagent は、次のタイミングで起動する。

1. 要求整理と初期調査の後、変更の中心と候補ファイルが見えた時点で、調査・実装 subagent を起動する。
2. 実装方針が見えた時点、またはテストを追加しそうになった時点で、テスト設計 subagent を起動する。実装 subagent が先に意味の薄いテストを書かないように、テスト方針は保留するか、テスト設計担当の判断と同期する。
3. 実装と検証が一通り終わり、完了報告やコミットに進む前に、完了前レビュー subagent を起動する。

subagent を早く起動しすぎて、要求理解や参照先の探索を丸投げしない。逆に、実装が終わってから初めてテスト設計 subagent を起動し、既に書いたテストの追認だけを求めてはならない。

### 依頼内容

調査・実装 subagent には、共有コンテキストに加えて次を渡す。

- 担当するファイル、module、責務
- 変更してよい範囲
- 既に読んだファイルと、追加で読むべき候補
- 期待する成果物。実装差分、実装案、調査結果のどれが必要かを明示する
- メインセッションが既に読んだ skill と、subagent 側でも読むべき skill
- 他の subagent やメインセッションの変更を戻さず、必要なら合わせること

テスト設計 subagent には、共有コンテキストに加えて次を渡す。

- 守るべき仕様、不安な仕様、壊れやすい境界
- 既存テストで既に守られていること
- 追加、修正、不要判断のどれを検討しているか
- `test-design-discipline` を使い、テストを追加するならどの誤変更を防ぐのか、追加しないなら何で担保するのかを答えること

完了前レビュー subagent には、共有コンテキストに加えて次を渡す。

- 最終差分の要約
- `git diff` 相当の確認対象。差分本文を渡すか、取得するためのコマンドと対象パスを指定する
- 実行した検証と結果
- テスト設計 subagent の判断
- 重点的に見てほしい不安点
- 指摘を MUST FIX、SHOULD FIX、確認済みの観点に分けること
- レビューだけを行い、ファイルを編集しないこと

subagent には必要な文脈を渡すが、全会話の要約を丸ごと渡さない。重い探索を避けるために、読ませたいファイルや差分が分かっている場合はパスを指定する。既に確認済みの事実を再調査させたい場合は、その理由を明示する。

### レビュー後の反復

完了前レビューで指摘が出たら、その項目だけを直して完了扱いにしない。修正によって別の漏れや設計の歪みが出ていないかを確認するため、修正後にもう一度レビューする。

反復の進め方:

1. レビュー指摘を MUST FIX、SHOULD FIX、今回扱わない指摘に分ける。
2. MUST FIX は原則として修正する。修正しない場合は、誤分類、今回の要求外、今回の要求外だが重要なので別作業化、ユーザー確認が必要な判断のどれに当たるのかを明示する。
3. SHOULD FIX は、今回の要求と因果関係があり、差分を不必要に広げないものだけ修正する。
4. 修正後、前回と同じ観点に加えて、修正箇所と影響範囲を重点に再レビューを依頼する。
5. 新しい MUST FIX が出たら、もう一度 1 から繰り返す。

反復を止めてよい条件:

- 未対応の MUST FIX がなく、新しい MUST FIX も出ない。
- 残った SHOULD FIX が、今回の要求に対して過剰、または別作業として扱うべきものだと説明できる。
- 同じ種類の SHOULD FIX が繰り返し出ており、直すほど差分が広がると判断できる。

反復を止める場合も、残した指摘と理由を完了報告に含める。2 回連続で新しい MUST FIX が出る、同じ種類の MUST FIX が再発する、または修正が今回の要求外へ広がる場合は、局所修正を続けず、方針変更、スコープ縮小、ユーザー確認のどれが必要かを決めてから再開する。

subagent を使わずに進めてよいのは、次のすべてを満たす場合だけ。

1. subagent 起動用 tool を探索した。
2. 利用可能な tool 一覧を確認しても subagent 起動用 tool が存在しなかった、または確認した tool の起動に失敗し、代替 tool も見つからなかった。
3. どの tool を確認し、どのように失敗し、代替 tool の有無をどう確認したのかをユーザーに報告した。
4. それでもメインセッションで続行する理由を明示した。

「実行環境により subagent を起動できませんでした」とだけ述べて作業を続けてはならない。確認していない場合は、subagent を使えないとは判断しない。

## 適正スコープ

差分の小ささではなく、要求に対して過不足のないスコープを優先する。

局所対応でよい場合:

- 表示文言、型の取り違え、条件分岐、設定値など、原因と修正箇所が明確に閉じている
- 既存の設計境界や責務分担に無理がなく、そこへ素直に差し込める
- 影響範囲を広げるほどリスクが増え、得られる保守性が小さい

根本的な対応を選ぶ場合:

- 同じ不具合や例外処理が複数箇所に散っている
- 呼び出し側ごとに同じ判断を繰り返しており、不変条件を表現できていない
- 既存の責務境界が原因で、局所対応するとさらに分岐や特殊ケースが増える
- データ構造、型、設定、API 境界を直すほうが、今後の変更を単純にできる
- ユーザー要求が個別症状ではなく、挙動や設計方針の変更を求めている

スコープを広げる場合も、現在の要求と因果関係がある範囲に限る。気づいた改善点があっても、今回の変更目的と強く関係しないなら現在の差分には混ぜない。必要に応じてユーザーに報告し、依頼があれば Issue 化や別作業として切り出す。

既存の設計意図が読み取れる場合は尊重する。既存スタイルと違う書き方を導入する場合は、その差分に明確な理由が必要。変更が複数の責務にまたがる場合は、責務ごとに差分を整理する。

## 抽象化判断

抽象化は、実際の重複、複雑さ、境界の曖昧さを減らす場合に追加する。

避けるもの:

- 将来使うかもしれないだけの汎用化
- 呼び出し側の都合を隠すだけの薄い wrapper
- 既存概念と並列に存在する新しい概念
- 1 回しか使わないのに理解コストが上がる trait や helper

望ましいもの:

- 不変条件を集約できる型
- 副作用境界を明確にする trait
- 再利用される純粋なデータ変換
- エラー処理やリトライなど、複数箇所で同じ判断が必要になる helper

## 検証方針

検証は「変更したから機械的にテストを足す」ではなく、「何が壊れると困るのか」を決めてから行う。

- 守るべき仕様、不安な仕様、壊れやすい境界を先に特定する
- テスト追加、テスト修正、テスト不要判断では、`test-design-discipline` を使う subagent に検証方針を確認させる
- テストを追加する場合は、そのテストがどの誤変更を防ぐのかを説明できるようにする
- テストを足さない判断をする場合も、型、既存テスト、ビルド、静的検査などで何が担保されるか確認する
- 完了前には、今回の変更の中心に対する検証があるかを確認する

## 作業の区切り

大きなタスクでは、ユーザーが毎回「コミットして」と言わなくても、意味のある単位が完了した時点で区切りを作るべきかを自律的に判断する。

区切りを検討する場面:

- 変更の中心となる実装が完了し、対応する検証も終わった
- 複数の責務にまたがる作業のうち、1 つの責務が独立して説明できる状態になった
- 大きめの作業で、ここまでの差分を安全な区切りとして残す価値がある
- PR 作成や別作業への移行前に、未コミットの変更を整理する必要がある

まだ区切らない場面:

- 実装の途中で、次の変更がないとビルドやテストが通らない
- 変更の意図をまだ説明できない
- 今回の目的と関係しない整理や試行錯誤の残骸が混ざっている
- ユーザーがコミットしない方針を明示している

区切りとしてコミットが適切なら、コミットの作法、ブランチ確認、メッセージ、署名、検証条件は `git-commit` スキルに従う。ここでは `git commit` の手順を重複して定義しない。

## 完了前の再確認

完了報告、コミット、PR の前に、最後の 1 回の確認で済ませない。差分を読んで「本当に要求を満たしているか」「もっと自然な境界がないか」を見直し、直す点があれば修正してからもう一度確認する。

1. ユーザー要求を読み直す。
2. 差分を読む。
3. 要求された挙動がすべて実装されているか確認する。
4. 変更の中心に対する検証があるか確認する。テスト追加、テスト修正、テスト不要判断が関わる場合は、`test-design-discipline` を使う subagent の確認結果を読む。
5. 既存コードの命名、分割、エラー処理、層分けとズレていないか確認する。
6. 追加した抽象化、依存関係、設定値が本当に必要か確認する。
7. 変更スコープが要求に対して過小でも過大でもないか確認する。
8. 実行すべき build / test / format / lint を実行する。
9. 完了前レビューを subagent に依頼し、差分、検証、スコープの見落としを確認する。
10. 漏れや違和感があれば修正し、レビュー後の反復ルールに従って再レビューする。
11. 反復を止める条件を満たしたら、修正した箇所、残した指摘、影響範囲を再確認する。

「一応動く」「軽く確認した」状態では完了扱いにしない。修正すべき点がなくなってから完了扱いにする。
